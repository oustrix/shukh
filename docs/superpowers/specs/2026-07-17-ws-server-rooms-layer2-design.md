## 1. Цель и рамки

**Цель.** Сетевой адаптер над `game.Session` (Слой 2, D-6): WebSocket-сервер, комнаты по
коду/ссылке, реконнект по HttpOnly-куке. Слой 2 отображает `соединение → PlayerID` и
`код комнаты → *game.Session`, дёргает синхронный API сессии и ретранслирует `Update` в браузеры.
Плюс — вынесенные из финального ревью Слоя 1 достройки Слоёв 0/1, без которых сеть неиграбельна
(в первую очередь тайм-аут голосования R-8.6).

**Критерий-девиз.** Слой 2 — **тонкий адаптер**: правила живут в движке (D-1), авторитетное
состояние и процедуры — в `game.Session` (Слой 1), а Слой 2 добавляет ровно транспорт: сокеты,
коды, токены, часы (таймеры), сборку мусора комнат. Никакой игровой логики в Слое 2.

**В рамках:**
- **B1. Достройки Слоёв 0/1** (аддитивно к влитому коду): `engine.CloseVote` + `VoteView` в
  проекции; `game.Session` — close-and-replace в `Subscribe`, системный `CloseVote()`, миграция
  хоста, `Snapshot`/`Restore`.
- **B2. Пакет `server/`** — `Hub` (реестр комнат + GC), `Room` (обёртка сессии + токены +
  таймеры), WS-соединение, протокол (DTO + кодек), шов хранилища (`RoomStore` + `MemStore`),
  генерация кодов и токенов, `Clock`-шов для тестируемости времени.
- **B3. Точка запуска** — `cmd/shukh-server/main.go`.

**Вне рамок (следующие спеки / отложено):**
- **Веб-клиент (`web/`, Слой 3)** — потребитель контракта; правится **другим агентом**. Слой 2
  файлы `web/` **не трогает**. Расхождения контракта фиксируются как задачи синхронизации для
  веб-стороны (см. §7.4), которые владелец передаст веб-агенту.
- **Персистентное хранилище** (БД/Redis) — MVP держит состояние in-memory; шов `RoomStore`
  заложен, чтобы добавить хранилище позже без переписывания (§4).
- **Общий тайм-аут хода / брошенная партия / фолд** — расширение OQ-2; в MVP не решается
  (см. §12 «Известные ограничения»). В рамках только тайм-аут **голосования** (R-8.6).
- **Аккаунты/история, серия партий (R-10.2)** — «позже» по дорожной карте.
- **Режим Culture (D-10)** — как и раньше, отложен.

## 2. Ключевые решения (для этого спека)

Нумерация `L2-n`, чтобы не пересекаться с `S-n` Слоя 1.

| # | Решение | Обоснование |
|---|---------|-------------|
| L2-1 | **Дедлок голосования R-8.6 чиним так:** таймер в Слое 2 → детерминированное действие `engine.CloseVote` резолвит разбор с текущими голосами. Формула прежняя: `противШУХа*2 > n` (большинство стола «Против ШУХа» → переклад на предъявителя Ш-8), молчание = «За ШУХ», никто не голосовал → подтверждение на цели. | Правило R-8.6 сохраняется дословно; кворум-логика остаётся в движке (D-1); движок чист — часы живут в транспорте (D-6/D-7). Молчащий/отключённый больше не морозит стол. `CloseVote` golden-тестируется без сети. |
| L2-2 | **Библиотека WebSocket — `github.com/coder/websocket`.** | Современный, zero-dependency, context-first, минимальный API; чисто ложится на `net/http`. Один модуль через **inline GOPROXY** (публичный прокси включается только в команде `go get`; глобальный дефолт не трогаем — см. память проекта). |
| L2-3 | **Хост ушёл из лобби → миграция хоста** на следующего по порядку входа; пустое лобби собирает GC. | Роль хоста нужна только в лобби (Start/SetConfig); в онлайне дисконнект хоста обычен. Тихая миграция «просто работает», дёшево (сдвиг указателя). |
| L2-4 | **Контракт доставки:** рендер **только из подписки**; возврат `Submit` = квитанция (ack/nack). `fanout` шлёт всем, включая подавшего. | Один путь рендера для всех сокетов; точно совпадает с веб-`Transport` (`send(action): void`, рендер из `subscribe`). Адаптер примитивен. |
| L2-5 | **Состояние — in-memory, но за швом `RoomStore`.** Дефолт `MemStore`; долговечная часть (код, конфиг, ростер, токены, `engine.State`, стадия) сериализуема; эфемерная (сокеты, каналы, таймеры) пересобирается. `Hub` зависит только от интерфейса. | MVP без БД (D-2: комнаты эфемерны, без аккаунтов). Шов делает подмену на БД/Redis правкой одной реализации + строки конструктора, без переписывания игры/протокола. Долговечная часть уже плоские данные (D-7) → снимок/восстановление тривиальны. |
| L2-6 | **Реконнект — по HttpOnly-куке, не по токену в URL.** Инвайт-ссылка несёт **только код** (`/r/CODE`, её шарят); личный токен выдаётся HTTP-шагом `join`, живёт в HttpOnly-куке пути комнаты. | Шаримая ссылка не должна содержать личное право на место (иначе скопировал адрес — отдал стул). HttpOnly → токен не читаем из JS (XSS-стойко), не попадает в URL/логи. **Ревизует D-2** («реконнект по токену в ссылке» → «по HttpOnly-куке»). |
| L2-7 | **Токен = случайный непрозрачный секрет (~128 бит), без JWT/подписи.** Сервер хранит соответствие `токен→PlayerID`. | Состояние и так на сервере; подпись не нужна. Знание токена = право на место (как ссылка на общий документ). |
| L2-8 | **`seed` тасовки — серверный** (crypto/rand), клиент его не выбирает. | Иначе клиент подберёт выгодную тасовку (чит). Детерминизм (D-7) сохраняется, источник seed — сервер; тесты инжектят точный seed/колоду. |
| L2-9 | **Время — за швом `Clock` (`Now`/`AfterFunc`).** Реальный в проде, фейковый в тестах. | Таймеры (voteTTL, grace, GC) тестируются детерминированно «прокруткой времени» без реальных `sleep`. |

## 3. Архитектура Слоя 2

Слой 1 (`game.Session`) знает **только** непрозрачный `PlayerID`. Всё сетевое — Слой 2. Он держит
две таблицы соответствий:

```
   код комнаты  ──►  *Room ──► *game.Session   (какая партия за кодом)
   токен        ──►  (комната, PlayerID)        (какое место за человеком)
```

### 3.1 Компоненты

- **`Hub`** — реестр комнат `код → *Room`. Создаёт комнаты, отдаёт по коду, собирает мусор
  пустых (через `RoomStore` и `Clock`). Единственная зависимость от хранилища — интерфейс
  `RoomStore`.
- **`Room`** — обёртка над одной `*game.Session`. Хранит код, таблицу `токен → PlayerID`, живые
  соединения, таймеры (голосования, grace, idle). После каждого изменившего состояние вызова —
  пишет снимок в `RoomStore` (write-through, §4).
- **WS-соединение (`conn`)** — на каждый сокет два потока: из браузера — разбор сообщений и вызовы
  `Session`; из подписки `Session` — сериализация `Update` в браузер.
- **HTTP-хендлеры** — создание комнаты, `join` (минтит токен + ставит куку), WS-апгрейд.

### 3.2 Раскладка пакетов

```
engine/                 Слой 0 — аддитивно: CloseVote, VoteView (§9). По-прежнему без сети/времени/RNG.
game/                   Слой 1 — аддитивно: close-and-replace, CloseVote(), миграция хоста,
                        Snapshot/Restore (§9).
server/                 Слой 2 — новый пакет:
  hub.go                реестр комнат + GC
  room.go               обёртка сессии + токены + таймеры + write-through
  conn.go               WS-соединение (два потока)
  protocol.go           DTO + кодек ↔ веб-контракт
  store.go              RoomStore + MemStore
  clock.go              Clock-шов
  token.go              генерация токенов
  code.go               генерация кодов комнат
cmd/shukh-server/
  main.go               точка запуска бинарника
```

## 4. Граница хранилища (`RoomStore`)

**Долговечное vs эфемерное:**

| Сорт | Что | Персистентно |
|------|-----|--------------|
| Долговечное (данные) | код, конфиг, ростер (`место↔PlayerID`+имя), `токен→PlayerID`, `engine.State`, стадия | да |
| Эфемерное (машинерия) | живые сокеты, каналы подписок, таймеры | нет — пересобирается при реконнекте/рестарте |

**Интерфейс (эскиз, имена уточняются в плане):**
```go
// Долговечный снимок комнаты — чистые данные, сериализуемо.
type RoomSnapshot struct {
    Code    string
    Tokens  map[Token]game.PlayerID
    Session game.SessionState // {Config, Host, Stage, Order, Names, Game engine.State}
}

// Шов хранилища. Hub зависит ТОЛЬКО от него.
type RoomStore interface {
    Save(RoomSnapshot) error
    Load(code string) (RoomSnapshot, bool, error)
    Delete(code string) error
    List() ([]string, error) // восстановление/GC
}

type MemStore struct { /* map под мьютексом */ } // дефолт MVP; реализует RoomStore
// Будущее: RedisStore / SQLStore — тот же интерфейс, остальное не трогаем.
```

**Write-through.** Во время игры `Room` работает с живой сессией в памяти; после каждого
изменившего состояние вызова — `RoomStore.Save(snapshot)`. С `MemStore` это дешёвая укладка
структуры; с будущим хранилищем — точка долговечности. Никто выше `Store` не знает, память там
или база.

**`game.Session.Snapshot()/Restore` (§9)** делают шов настоящим: сессия умеет выгрузиться в
`SessionState` (глубокая копия, без аллиасинга живых мап) и восстановиться со свежими пустыми
подписками.

## 5. Комнаты: код, жизненный цикл, сборка мусора

### 5.1 Код комнаты
Короткий человеко-передаваемый: **6 символов** из алфавита без визуально-неоднозначных
(без `O/0/I/1/L`, ~32 символа). При создании `Hub` генерит код и проверяет коллизию по реестру
(перегенерит при совпадении). Верхний регистр; на вводе нормализуем.

### 5.2 Жизненный цикл
```
создана(Lobby) ──► Playing ──► Finished
      │              │             │
      └──── все соединения мертвы дольше grace ────►  GC (Hub.Delete + Store.Delete)
```

### 5.3 Сборка мусора и grace
Комната удаляется, когда:
- **все соединения мертвы** дольше **grace-периода** (дефолт ~5 мин — пережить обрыв и реконнект),
  **или**
- партия **Finished** и прошёл короткий **idle-TTL** (дефолт ~1–2 мин).

Механика — фоновый свипер / пер-рум таймеры через `Clock`. Всё эфемерное, к движку отношения нет.

### 5.4 Реконнект-grace (связь с §6)
Обрыв сокета **не** ведёт к немедленному `Leave`: `Room` держит место, гасит подписку, взводит
grace-таймер, помечает игрока «отключён». Реконнект в пределах grace — отменяет таймер и
переподключает. Истёк grace — в лобби `Leave` (с миграцией хоста L2-3); в игре место остаётся
«отключённым» (см. §12).

## 6. Идентичность и реконнект

### 6.1 Две «ссылки», их нельзя путать
- **Инвайт-ссылка** (шарят друзьям): **только код** — `/r/CODE`, без токена. Все открывают одну.
- **Личная идентичность места**: **не в URL вообще** — HttpOnly-кука.

### 6.2 Токен и его связь с `PlayerID`
При входе игрока `Room` генерит `PlayerID` (внутренний id места) и `Token` (случайный секрет
~128 бит, base64url), кладёт в `токен→PlayerID`. Токен — в HttpOnly-куке пути комнаты;
`PlayerID` наружу не отдаётся.

### 6.3 Поток входа/реконнекта
```
POST /r/CODE/join {name}   →  Session.Join(PlayerID, name); минт PlayerID+Token;
                              Set-Cookie(HttpOnly, SameSite, Secure-в-проде, path=/r/CODE); 200
GET  /r/CODE  (WS-апгрейд)  →  браузер сам шлёт куку → сервер узнаёт место → Subscribe
обрыв                       →  Room держит место + grace-таймер (§5.4)
реконнект                   →  переоткрыть WS; кука на месте → узнан → Subscribe (close-and-replace)
grace истёк                 →  лобби: Leave (миграция хоста); игра: место «отключено»
```
Тонкость: куку ставим на HTTP-ответе `join` (имя приходит там), **не** на WS-рукопожатии — на
апгрейде игрок ещё не представился. Клиентский JS токен **не видит и не хранит** (ни URL, ни
localStorage).

### 6.4 close-and-replace в `Subscribe`
Повторный `Subscribe(id)` (реконнект при ещё-живом старом сокете) сейчас **сиротит** первый канал
(утечка горутины) и делает старый `cancel` пустышкой. Правка (§9): перед установкой нового ящика
закрыть старый — старый работник получает «закрыто» и штатно выходит.

## 7. Протокол WS

### 7.1 Конверт
Каждое сообщение — JSON с дискриминатором `type`.

**Клиент → сервер:**
| `type` | payload | Слой 1 |
|--------|---------|--------|
| `action` | `{action: <Action>, reqId?}` | `Session.Submit(playerID, action)` |
| `setConfig` | `{config}` | `Session.SetConfig(host, cfg)` (только хост) |
| `start` | — | `Session.Start(host, seed)` (seed серверный, L2-8) |
| `leave` | — | `Session.Leave(playerID)` |

`<Action>` — **полный** набор действий движка как размеченный union: `playCard`,
`takeBottomAndPass`, `podkladkaWest`, `claimShukh`, `giveShukhCard`, `takeShukhCards`,
`declareOneCard`, `askCount`, `askAboutWest`, `discardWest`, `claimSubjective`, `vote`.
**`closeVote` в клиентский набор НЕ входит** — системное действие по таймеру (L2-1).

**Сервер → клиент:**
| `type` | payload | смысл |
|--------|---------|-------|
| `update` | `{you, roomCode, stage, roster, view\|null, legal, events, voteDeadline?}` | сериализованный `game.Update` + мета; единственный путь рендера |
| `ack` | `{reqId}` | «действие принято» |
| `error` | `{reqId?, code, message}` | «отклонено»: `IllegalAction`/`game.*` → код |

`you` — индекс своего места (нужен в лобби, где `view == null`). `voteDeadline` (unix-ms) — мета
Слоя 2 для обратного отсчёта голосования (§8), **не** в `engine.SeatView`.

### 7.2 Бюллетень голосования — явные формулировки
На границе протокола голос называется явно: **«За ШУХ»** (ШУХ справедлив → подтвердить на цели)
и **«Против ШУХа»** (ШУХ бракованный → переклад на предъявителя как Ш-8). Движковое поле
`Vote.Support` (`true` = «Против ШУХа») **остаётся как есть** — переименование отревьюенного
движка без пользы; кодек протокола мапит явные имена ↔ `Support`.

### 7.3 Кодек (W-3)
`engine.Action`/`engine.Event` — запечатанные Go-интерфейсы; ручной кодек `Go ↔ размеченный JSON`
(масти глифами `♠♥♦♣`, ранг числом), зеркалящий TS-юнионы `web/src/contract/types.ts`.
Дисциплина — та же W-3 «править вручную по обе стороны».

### 7.4 Расхождения с текущим веб-контрактом (задачи для веб-агента — НЕ трогаем `web/`)
Фиксируются здесь; владелец передаёт их веб-агенту на синхронизацию:
1. Веб-`Action` union **неполон**: нет `claimSubjective`, `vote`, `declareOneCard`, `askCount`,
   `askAboutWest`, `discardWest`.
2. Веб-`GameSnapshot` **без `legal`**; Слой 1 `Update` его несёт — фронту поле нужно добавить.
3. `VoteView` в проекции (§8) — новое поле, фронту добавить для UI голосования.

## 8. Голосование по сети

Собирает L2-1 (тайм-аут), item 4 (контекст в проекции), OQ-3 (онлайн-UX).

### 8.1 Таймер и `CloseVote`
- **Движок:** `engine.CloseVote` (§9) — резолвит открытый разбор с текущими голосами; no-op при
  закрытом.
- **Session:** системный `CloseVote()` (§9) — мимо `authorize`, берёт мьютекс, применяет, фанаутит.
- **Room:** держит часы через `Clock`:
  ```
  видит VoteOpened   →  взвести таймер voteTTL (дефолт ~30с, конфигурируемо)
  видит VoteResolved →  погасить таймер
  таймер сработал    →  Session.CloseVote()
  ```

### 8.2 Гонка «голос vs таймер» безопасна
Всё сериализуется `Session.mu`. Последний голос добил явку до 100% → авто-резолв → таймер стреляет
на `Adjudication == nil` → no-op. Таймер первым → резолв → опоздавший `Vote` отвергается `isLegal`
как нелегальный (Room отдаёт `error`, безвредно).

### 8.3 Контекст для переподключившегося (item 4)
В `engine.SeatView` — опциональная сводка:
```go
type VoteView struct {
    Claimant SeatID
    Target   SeatID
    Code     ShukhCode
    Voted    []SeatID  // кто уже проголосовал (только факт, НЕ как)
}
// SeatView: Vote *VoteView  // nil без разбора
```
Течёт через `game.Update` автоматически. Переподключившийся видит «кто-кого-за-что» и кто уже
отдал голос.

### 8.4 Тайна бюллетеня
До резолва показываем **факт** голоса (кто отдал), но **не содержание**. На резолве исход виден
(`VoteResolved`). Проще и снимает соц-трение посреди разбора.

### 8.5 Онлайн-UX (OQ-3)
- Избиратели — весь стол `0..n-1`, включая вышедших (R-8.9/R-9.5) — уже в движке (`voterEligible`).
- Кворум вышедших — тоже в движке.
- Сигналы клиенту: контекст из `VoteView` + обратный отсчёт из `voteDeadline` (мета Слоя 2).

## 9. Правки Слоёв 0/1 (радиус поражения)

Движок и `game` влиты с ревью «ready to merge»; достраиваем **аддитивно** (дорожная карта отложила
эти пункты сюда). Движок остаётся чистым (без времени/сети): `CloseVote` детерминирован, `VoteView`
— данные.

**Слой 0 (`engine/`):**
1. **`CloseVote`** — действие без сиденья. `isLegal`: только при `Adjudication != nil`; `Apply`:
   зовёт существующий `resolveAdjudication` с текущими голосами; при закрытом разборе — безвредный
   no-op. Golden-тесты.
2. **`VoteView` + `SeatView.Vote *VoteView`** — сводка разбора в проекцию, заполняется в `View()`;
   чистая, без аллиасинга.
3. *Не трогаем* `Vote.Support` (оставляем + уточняем doc-комментарий).

**Слой 1 (`game/`):**
4. **`Subscribe` → close-and-replace** (утечка item 2).
5. **`Session.CloseVote()`** — системный вход для таймера (мимо `authorize`).
6. **Миграция хоста** в `Leave` при уходе хоста из лобби (L2-3).
7. **`Session.Snapshot() SessionState` + `game.Restore(SessionState)`** — шов хранилища.
   `SessionState = {Config, Host, Stage, Order, Names, Game engine.State}`, глубокая копия, `Restore`
   — свежие пустые подписки.
8. *Контракт Submit/фанаут (L2-4):* код не меняем (fanout уже шлёт всем, Submit уже возвращает
   события) — только **doc-комментарий**, пиннящий «возврат = ack-эхо, Слой 2 не ре-рендерит», +
   дисциплина адаптера. Сигнатура без изменений.

## 10. Обработка ошибок

- **Нелегальное действие/отказ Session** (`IllegalAction`, `game.*`) → маппится в код и уходит
  **только инициатору** как `error{reqId, code, message}`. Состояние не тронуто, фанаута нет
  (гарантия Слоя 1).
- **Slow consumer** — как в Слое 1: подписка буферизована, отправка неблокирующая; переполнение
  дропает дельту, клиент восстанавливается `Snapshot`. Игра под `Session.mu` не блокируется.
- **Обрыв сокета** — не ошибка: grace-таймер (§5.4).
- **Идентичность на WS-апгрейде:** нет куки → клиент направляется в `join` (ввод имени). Кука с
  токеном не из карты комнаты (протух / чужая комната / GC) → `error{code: "seatNotFound"}`, клиент
  падает в `join`. (Единое правило, без второго варианта.)
- **Никаких паник** в штатных путях; нарушение инварианта §11 в движке — баг, не маскируем.

## 11. Тестирование

Стиль проекта: детерминизм, точная колода, in-memory, `-race`, фейковый `Clock`.

**Слой 0 (`engine/`) — golden:**
- `CloseVote`: резолв при частичной явке (большинство «Против ШУХа» → Ш-8; недобор/ничья/ноль →
  подтверждение на цели); no-op при закрытом разборе; инварианты §11.
- `VoteView`: заполнение верно, `nil` без разбора, чистота.

**Слой 1 (`game/`):**
- `Subscribe` close-and-replace: старый ящик закрыт, горутина не течёт (`-race`).
- `CloseVote()`: резолвит; гонка «голос vs таймер» безопасна.
- Миграция хоста при `Leave` хоста из лобби.
- `Snapshot`/`Restore`: round-trip идентичен; глубокая копия (мутация оригинала не течёт в снимок).

**Слой 2 (`server/`):**
- Юниты in-memory: `Hub` create/get/GC; генерация кода + коллизия; `MemStore` CRUD;
  `токен→PlayerID`; кодек round-trip (Go↔JSON против ожидаемых форм веб-контракта: глифы, юнионы).
- Интеграция: `httptest`-сервер + WS-клиент `coder/websocket`; полный сквозняк
  `создать → join → start → скриптованная партия → открыть голосование → таймер → резолв → finish`;
  реконнект (обрыв → кука → снапшот → продолжить); double-connect (close-and-replace);
  slow-consumer не блокирует.
- Время — через фейковый `Clock` («прокрутка» вместо `sleep`).

**Явно вне тестов:** правки `web/` (другой агент); персистентное хранилище (только `MemStore`).

## 12. Известные ограничения (в MVP не решаем; записаны, чтобы не забыть)

1. **Рестарт сервера роняет текущие партии** и **один инстанс** (in-memory, L2-5). Шов `RoomStore`
   позволяет добавить персистентность/масштаб позже без переписывания.
2. **Брошенная в свой ход партия морозит стол.** Grace спасает только от обрывов; навсегда-ушедший
   игрок на своём ходу блокирует стол. Общий тайм-аут хода / авто-ход / фолд — будущая работа
   (расширение OQ-2). В MVP — «партия испорчена, пересоздайте комнату».
