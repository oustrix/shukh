# Спец 1 — Ядро движка «Шух» (Слой 0)

> **Статус.** Черновик 1 (2026-07-15). Проектный документ первого спека дорожной карты
> (см. [`docs/architecture.md`](../../architecture.md)).
>
> **Отношение к другим документам.** Правила — источник истины
> [`docs/shukh-rules.md`](../../shukh-rules.md) (ссылки `R-§.n`, `Ш-n`, `I-n`, `V-n`).
> Архитектурные решения `D-n` — в [`docs/architecture.md`](../../architecture.md).

---

## 1. Цель и рамки

**Цель.** Чистое детерминированное **ядро игры** на Go (Слой 0): вся логика правил «Шух»
как функция над состоянием, без сети, I/O, времени и внутренней случайности. Ядро — это
авторитетный «судья»: оно знает полное состояние, перечисляет легальные ходы, применяет
действия и проецирует видимость на каждого игрока.

**В рамках Спеца 1:**
- Модель состояния и зон (I-1).
- Карты для колод 36/52, старшинство и бой (§3).
- Автоматическая раздача (§4, кроме §4.3).
- Жизненный цикл кона (§5): заход, бой, взятие, закрытие, отбой, очередь.
- «Одна карта» (§6), выход игрока и эндшпиль `6(2)♥` (§9.1, §9.2).
- Механика ШУХа (§7–§8) в режимах энфорсмента **Guard** и **Middle** (D-10).
- Детерминизм и события (D-7).
- Тесты против инвариантов §11.

**Вне рамок Спеца 1 (позже):**
- Режим **Culture** (D-10) — требует отката «неправильной» позиции.
- Раздачные ШУХи §4.3 — при авто-раздаче Ш-1 не возникает (мёртвая ветка).
- Опциональные варианты §12 (флаги в `RuleSet` присутствуют, но выключены и не реализованы).
- Всё сетевое/UI (Слои 1–3) — отдельные спеки.

**Критерий-девиз.** Движок обязан быть «железным»: любой инвариант §11 держится после
любого применённого действия; нелегальные (в текущем режиме) действия отвергаются
типизированной ошибкой, а не порчей состояния.

## 2. Модель состояния и зоны

Инвариант I-1: каждая карта колоды в любой момент — ровно в одной зоне.

```go
type State struct {
    Rules  RuleSet             // размер колоды + флаги вариантов §12
    Mode   EnforcementMode     // Guard | Middle (| Culture — позже)
    Seats  []SeatID            // порядок по часовой (R-2.13), фикс на партию
    Phase  Phase               // Playing | Finished (раздача атомарна в NewGame)

    Talon   []Card             // колода; после NewGame пуста (раздача §4 завершена)
    Hands   map[SeatID][]Card  // рука игрока; в игре — множество (порядок неважен)
    Table   []Card             // кон: стек снизу→вверх; [0]=нижняя, [len-1]=верхняя
    Discard []Card             // отбой, закрыт (R-2.9)
    Shukh   map[SeatID][]Card  // отложенные ШУХ-карты, «рубашкой вверх» (R-2.10, I-3)

    Turn   SeatID              // чья очередь
    Live   map[SeatID]bool     // «живые» игроки (R-5.5.1, R-9.1)
    Finish []SeatID            // порядок выхода → места (R-9.2)

    Endgame EndgameState       // §9.2: достигнут ли режим «2 игрока», спрашивали ли про 6(2)♥
    Pending *Adjudication      // активное предъявление/голосование (R-8.6), если есть

    Unsettled *Unsettled       // Middle: открытое окно немедленной поимки последнего действия (§6.1)
}

type RuleSet struct {
    DeckSize       int  // 36 | 52
    PodkladkaSnizu bool // V-1…V-4 — MVP: false, не реализовано
    Jokers         bool // §12 — MVP: false, не реализовано
}

type EnforcementMode int // Guard, Middle (дефолт), Culture(позже)
```

**Карты — параметрически (R-2.3).** `Rank` — порядковый индекс в активной колоде; «старше»
(R-3.5) и «+1» с замыканием Туз→младшая (R-4.5) считаются относительно `Rules.DeckSize`.
Спецкарты — предикаты от правил, не «магические» значения:
`IsTrump` (♦), `IsLowestHeart` (`6(2)♥`), `IsSecondLowestHeart` (`7(3)♥`),
`IsQueenHearts`, `IsStarter` (`9♦`).

## 3. Старшинство и бой (§3)

Матрица легального боя верхней карты `T` картой `C` (R-3.1, R-3.5 — строго по номиналу):

| Верхняя `T` | Чем можно бить `C` |
|---|---|
| ♠ пика | только **старшей ♠** (козырь ♦ не бьёт — R-3.3, I-7) |
| ♦ бубна (козырь) | только **старшей ♦** |
| ♥ черва / ♣ трефа | **старшей той же масти** ИЛИ **любой ♦** (козырь любого номинала, R-3.2) |

- Крыть не-козырём **чужой** масти нельзя (R-3.4). Обязанности «сначала в масть» нет.
- **`6(2)♥`** (R-3.6): бить ею нельзя (младшая). Уйти можно только заходом (R-3.6.1) или
  западло-подкладкой под `7(3)♥` (R-3.6.2).
- **`Дама ♥`** (R-3.7): бьёт **любую** карту и **немедленно** отправляет кон в отбой
  (R-3.7.1, минуя счёт R-5.5). Ходить с неё нельзя (R-3.7.2 → Ш-2).

## 4. API движка

```go
func NewGame(cfg Config, seed int64) (State, []Event)
func LegalActions(s State, seat SeatID) []Action
func Apply(s State, a Action) (State, []Event, error) // чистая; не мутирует вход
func View(s State, seat SeatID) PlayerView
func CheckInvariants(s State) error                   // §11; в тестах после каждого Apply
```

- **`NewGame`** тасует по `seed` (Fisher–Yates) и атомарно проводит раздачу §4, возвращая
  готовые руки и события. `cfg` = `{RuleSet, EnforcementMode, []Player}`.
- **`LegalActions`** перечисляет легальные для `seat` действия при текущем режиме — для
  UI-подсказок (гасить нелегальное) и как исполняемая спецификация правил.
- **`Apply`** — чистый редьюсер: новое состояние + события, либо `*IllegalAction{Code, Rule}`.
  Вход не мутируется (copy-on-write). `panic` только на нарушение внутреннего инварианта (баг).
- **`View`** (D-9): своя рука целиком; у соперников — только счётчик карт; `Table`, счётчики
  `Discard`/`Shukh`, `Turn`, `Live` — публично; содержимое `Shukh`/`Discard` скрыто.

## 5. Действия

**Ход в свою очередь (R-5.3):**
```go
PlayCard{Card}       // заход в пустой кон (R-5.2, кроме Дамы♥) ИЛИ бой верхней (§3)
TakeBottomAndPass{}  // R-5.3b/R-5.8: взять единственную нижнюю карту кона, спас
PodkladkaWest{}      // R-5.3c: западло 6(2)♥ под 7(3)♥ (R-3.6.2), когда применимо
```
`PlayCard` различает заход и бой по состоянию `Table` (пуст → заход).

**Социальные/процедурные (вне своей очереди):**
```go
DeclareOneCard{}               // §6: объявление «Одна карта!»
AskCount{Target}               // «Сколько карт?» → авто-проверка Ш-11 (R-6.2, R-6.3)
AskAboutWest{Target}           // эндшпиль §9.2: «есть 6(2)♥?» → авто-проверка Ш-12
ClaimShukh{Target, ReasonCode} // предъявить ШУХ
Vote{ClaimID, Support}         // голосование по оспариванию (R-8.6)
TakeShukhCards{}               // взять отложенные ШУХ-карты (только после закрытия кона, R-8.3)
```

## 6. Раскладка ШУХов и режимы энфорсмента (D-1, D-10)

Каждый триггер ШУХа обрабатывается одним из трёх способов; выбор зависит от режима.

| ШУХ (§7) | Природа | Guard | Middle (дефолт) | Culture (позже) |
|---|---|---|---|---|
| нелегальный бой §3/I-7; карты нет в руке | ломает целостность | БЛОК | БЛОК | БЛОК (если карты нет) |
| Ш-7 ход не в очередь | целостность | БЛОК | БЛОК | ловится «ШУХ!» |
| Ш-2 заход с Дамы♥ | детектируемо | БЛОК | ловится «ШУХ!» | ловится «ШУХ!» |
| Ш-3 взял ШУХ рано | детектируемо | БЛОК | ловится «ШУХ!» | ловится «ШУХ!» |
| Ш-4a/4b, Ш-5 тайминг отбоя | детектируемо | авто-отбой (не возникает) | ловится «ШУХ!» | ловится «ШУХ!» |
| Ш-11 не сказал «одна карта» | детектируемо по `AskCount` | авто-суд | авто-суд по `AskCount` | авто-суд по `AskCount` |
| Ш-12 эндшпиль `6(2)♥` | детектируемо по `AskAboutWest` | авто-суд | авто-суд | авто-суд |
| Ш-1 касание колоды в раздаче | — | N/A (авто-раздача) | N/A | N/A |
| Ш-6 завис, Ш-8 ложный ШУХ, Ш-9 «ШУХ» зря, Ш-10 небрежность | субъективно | предъявление+голосование | то же | то же |

**Механика подтверждённого ШУХа** (§8, применяется одинаково, авто или по голосованию):
- R-8.1: каждый игрок даёт провинившемуся **по одной** карте; **не последнюю** (R-8.1.1, I-2).
- R-8.2: карты кладутся в его ШУХ-зону (`Shukh`), не в руку (I-3).
- R-8.3: поднять их в руку можно только по **закрытии текущего кона**; раньше → Ш-3.
- R-8.6: любой ШУХ оспорим; при поддержке **большинства** — перекладывается на предъявителя (Ш-8).
- R-8.5: после начисления/оплаты корректная позиция восстановлена, затем игра продолжается.

`Adjudication` (`Pending`) моделирует активное голосование: кто предъявил, кому, повод,
голоса, кворум большинства стола (в т.ч. вышедшие — R-8.9). Судейство/голоса — процедура;
движок не «оценивает» субъективную суть, лишь считает голоса и применяет исход.

### 6.1. Модель поимки и отката (Middle vs Culture)

Ключ к тому, чтобы Middle был реализуем без тяжёлой машинерии:

- **Middle — поимка только немедленная.** «Спорное» действие (заход с Дамы♥ Ш-2, ранний
  забор ШУХа Ш-3, тайминг отбоя Ш-4/Ш-5) можно предъявить **до хода следующего игрока**.
  Поэтому спорное действие всегда — **последнее** применённое, и восстановление позиции
  (R-8.5) — это **локальный детерминированный реверс одного действия** (напр. вернуть Даму♥
  со стола в руку), а не откат последовательности ходов. После разрешённого-нелегального
  действия состояние помечается **неустойчивым** (`Unsettled`) до закрытия окна поимки; пока
  окно открыто, следующий ход блокируется, а любой игрок может `ClaimShukh`. Если не
  предъявили — окно закрывается, состояние становится устойчивым, действие «прижилось»
  (R-1.4.1). Само окно/тайминг во времени — забота Слоя 1–2; движок лишь различает
  устойчивое/неустойчивое состояние и умеет реверснуть последнее действие.
- **Culture — поимка отложенная** (R-1.4.1: «если заметили», в т.ч. спустя ходы) → требует
  **общего отката/реплея** последовательности → вынесено за рамки Спеца 1.

Следствие для инвариантов: §11 держатся на **устойчивых** позициях. В неустойчивом
состоянии (Middle, открытое окно поимки) I-6/I-7 могут быть временно ложны — `CheckInvariants`
в этот момент не ассертится (см. §10).

## 7. Раздача (§4, автоматическая)

`NewGame` детерминированно проводит раздачу по R-4.3…R-4.10:
- Стопка каждого игрока — стек, значима верхняя карта; правило «+1» (R-4.3) с замыканием
  Туз→`6(2)` (R-4.5); конфликт адресатов — первому по часовой (R-4.6).
- Ход раздачи: разгрузка своей стопки → тяга → адресация сопернику/себе (R-4.4).
- Итог — стартовые руки (R-4.10). §4.3 не применяется (Ш-1 при авто-раздаче невозможен).

*Замечание.* Внутренняя реализация раздачи как отдельная детерминированная процедура;
её пошаговость наружу не выставляется (клиент получает готовую руку — D-9).

## 8. Кон, очередь, выход, эндшпиль

- **Заход (R-5.1, R-5.2):** первый кон партии — владелец `9♦`; далее — по R-5.7. Заходят
  любой картой, кроме Дамы♥.
- **Закрытие (R-5.5):** число карт на кону = числу живых игроков; порог падает с выходом
  игроков (R-5.5.1). Дама♥ закрывает немедленно (R-3.7.1). Иных способов нет (I-4).
- **Отбой (R-5.6):** закрывший убирает кон. В Guard — авто; в Middle — действие, чей
  неверный тайминг ловится (Ш-4/Ш-5).
- **Кто заходит следующим (R-5.7):** закрывший кон; после западла — съевший кон (R-5.7.1).
- **Взятие нижней (R-5.8):** уменьшает счёт кона; порог закрытия проверяется по актуальному
  числу карт.
- **Выход (R-9.1):** рука пуста И ни одной карты игрока нет в незакрытом коне. Пустая рука,
  но карта висит в коне (R-5.9) — игрок берёт нижнюю и продолжает.
- **Эндшпиль на 2 (§9.2):** `6(2)♥` подлежит сбросу в отбой (R-9.3). Порядок R-9.4:
  обладатель может сбросить безнаказанно, пока его не спросили (`AskAboutWest`); спросили и
  она у него → Ш-12; использовал не скинув → Ш-12 + пропуск хода + обязан сбросить.
- **«Одна карта» (§6):** `DeclareOneCard` фиксирует объявление; `AskCount` проверяет Ш-11 с
  учётом R-6.3 (успел походить — ШУХа нет). В счёт входят только карты руки, не ШУХ-зона (R-6.4).

## 9. Детерминизм и события (D-7)

- Единственный источник случайности — `seed` (тасовка). Ни времени, ни `rand` внутри →
  воспроизводимость (тесты, отладка, реплеи).
- События (`CardPlayed`, `ConClosed`, `ConSwept`, `ShukhAssessed`, `ShukhPaid`,
  `CardsTaken`, `PlayerFinished`, `OneCardDeclared`, `VoteOpened`, `VoteResolved`, `TurnPassed`,
  …) — источник для анимаций/лога на верхних слоях. Фильтрацию скрытой инфы под игрока делает
  Слой 1 через `View`-правила, не движок.

## 10. Тесты

- **Инварианты §11 как оракул:** `CheckInvariants` (I-1 сохранение карт, I-2 не отдаём
  последнюю, I-4/I-6 закрытие и Дама♥, I-5 отбой не возвращается, I-7 пики) — после каждого
  `Apply`, приводящего к **устойчивой** позиции. В неустойчивом состоянии (Middle, открытое
  окно поимки — §6.1) проверяются лишь всегда-истинные инварианты (I-1 сохранение карт);
  I-6/I-7 ассертятся после закрытия окна/реверса.
- **Fuzz-проигрыши:** сидированные случайные *легальные* партии (через `LegalActions`) до
  конца; инварианты на каждом шаге; партия терминируется с валидным `Finish`.
- **Golden-сценарии:** микро-пример раздачи (R-4.10), западло (R-3.6.2), закрытие Дамой♥
  (R-3.7.1), эндшпиль `6(2)♥` (§9.2), по одному пути на каждый ШУХ и на оспаривание (R-8.6).
- **Табличные тесты** матрицы боя (§3) для колод 36 и 52, включая упрямость пик (I-7) и
  безноминальность козыря (R-3.2).
- **Режимы:** ключевые ШУХ-пути прогоняются в Guard и Middle (разное поведение по D-10).

## 11. Внутренние итерации

1. **Карты + бой §3:** колода 36/52, `RuleSet`, предикаты старшинства/«+1», матрица боя,
   каркас I-7/I-4. Тесты: матрица боя.
2. **Раздача §4 + `NewGame` + I-1:** авто-раздача, сохранение карт. Тесты: микро-пример, fuzz-раздача.
3. **Жизненный цикл кона §5 + выход §9.1:** заход/бой/взять/закрытие(счёт+Дама♥)/отбой,
   очередь. Тесты: fuzz-проигрыш до терминации.
4. **ШУХи §7–§8 (Guard+Middle) + «одна карта» §6 + эндшпиль §9.2:** блок-набор,
   ловимый набор, авто-суд (Ш-11/Ш-12), предъявление/голосование. Тесты: каждый путь ШУХа.

## 12. Критерии готовности (Definition of Done)

- Все четыре итерации реализованы; публичный API §4 стабилен.
- `CheckInvariants` не падает ни в одном тесте; fuzz гоняет ≥ N сидированных партий до
  терминации без нарушений (N задаём в плане).
- Матрица боя §3 покрыта для обеих колод; все golden-сценарии зелёные.
- Режимы Guard и Middle покрыты на ключевых ШУХ-путях.
- Ноль `panic` на любой вход `Apply` (только типизированные ошибки).
- Движок не импортирует сетевых/I/O-пакетов (Слой 0 чист).

## 14. Решения итерации 3 (жизненный цикл кона §5 + выход §9.1)

Итерация 3 реализует «счастливый путь» лайфсайкла в режиме **Guard**: всё детектируемое либо
гасится в `LegalActions`, либо разрешается автоматически (авто-отбой) — ШУХов не возникает. Ниже
зафиксированы решения, принятые при разборе острых углов; ШУХи §7–§8, «одна карта» §6, эндшпиль
§9.2, Middle-окно `Unsettled` и `View` — вне итерации 3.

### 14.1. Модель стола: владелец у карты

`Table` перестаёт быть `[]Card` и становится `[]TableCard{Card; By SeatID}` — у каждой карты кона
явный владелец. Это нужно для выхода: R-9.1/R-5.9 требуют знать, есть ли карта игрока в **незакрытом**
коне, а R-5.8 «взять нижнюю» отдаёт `Table[0]` в руку конкретного игрока и может обнулить его
присутствие в коне. Существующие чтения `Table` (напр. `CanBeat` по верхней карте) правятся под
`TableCard`. I-1 (`CheckInvariants`) считает `tc.Card` каждой `TableCard`.

### 14.2. Действия и `LegalActions`

- `PlayCard{Card}` — заход в **пустой** кон (любая карта, кроме `Дамы ♥`, R-5.2) **или** бой верхней
  карты по §3 (`CanBeat`). Заход/бой различаются по `len(Table)`.
- `TakeBottomAndPass{}` — взять `Table[0]` в руку и пропустить (R-5.3b/R-5.8). Доступно всегда при
  непустом коне; для **безрукого живого** игрока (R-5.9) — единственное доступное действие.
- `PodkladkaWest{}` — западло: `6(2)♥` из руки под низ кона, когда `Table[0]` = `7(3)♥` (R-3.6.2).

**Инвариант ядра:** `Turn` всегда указывает на место с ≥ 1 легальным действием, либо
`Phase == Finished`. Единственная «нулевая» ситуация (пустой кон + в руке только `Дама ♥`)
разрешается скипом (см. 14.4). Ассертится в fuzz.

### 14.3. Конвейер `resolve` (после каждого действия)

Порядок пост-обработки детерминирован. Ключевой принцип: **любое опустошение стола** (закрытие+отбой
или западло-съедание) запускает проверку выхода **всех** безруких живых (их карт больше нет в открытом
коне); «взять нижнюю» — проверку выхода **только** владельца снятой нижней карты.

1. **Заход** (стол был пуст → 1 карта): закрытия нет (порог ≥ 2), очередь → следующему живому.
2. **Бой**: `Дама ♥` → закрытие немедленно (R-3.7.1, минуя счёт); иначе `len(Table) == liveCount`
   → закрытие по счёту (R-5.5); иначе очередь дальше.
3. **Закрытие**: снос всего кона в отбой (авто, R-5.6) → выход всех безруких живых → заходящий =
   закрывший; если он вышел — следующий живой по часовой (**R-5.7.2**) → `Turn = заходящий`.
4. **Взять нижнюю**: снять `Table[0]`; если владелец безрук — проверить его выход; закрытие
   невозможно (взятие лишь уменьшает счёт); пустой стол → следующий заходит; очередь дальше.
5. **Западло**: весь кон (вкл. подложенную `6(2)♥`) → в руку следующего живого (едок); едок
   заходит (R-5.7.1, при выходе едока — R-5.7.2); выход прочих безруких живых; отбоя нет.
6. **Терминация**: партия кончается при `liveCount ≤ 1`. `== 1` → он проигравший (последнее место,
   R-10.1). `== 0` (последние вышли одновременно на одном закрытии, R-10.1.1) → места по порядку
   одновременных выходов (по часовой от закрывшего), последний — проигравший. `Finish` — полный
   рейтинг мест; порядок одновременных выходов — по часовой от закрывшего.

### 14.4. Заход `Дамой ♥`: скип — поведение, специфичное для Guard

Заход с `Дамы ♥` запрещён (R-3.7.2). В **Guard** это блокируется. Достижим вынужденный дедлок:
кон пуст, в руке игрока осталась **только** `Дама ♥` — заходить нечем. Разрешение (**только для
Guard**):

- Если единственно возможный ход живого игрока — запрещённый заход `Дамой ♥` (кон пуст, рука =
  `{Дама ♥}`), его **ход пропускается** (событие `TurnSkipped{Seat}`); `Дама ♥` будет сыграна боем,
  когда появится что крыть. Это сохраняет I-6 (`Дама ♥` никогда не заходная) и терминацию (в колоде
  одна `Дама ♥` → скипаем максимум одного; следующий живой всегда может зайти).

> **Это приём режима Guard, а не правило игры.** В **Middle/Culture** (итерация 4) заход `Дамой ♥`
> **разрешён и ловится как Ш-2** («ШУХ!»), поэтому дедлока там нет и скип не применяется. За живым
> столом ситуация разрешается через Ш-2, а не пропуском хода.

### 14.5. Инварианты, добавляемые в `CheckInvariants`

- **I-6**: `Дама ♥` не присутствует в `Table` устойчивого состояния (она закрывает кон немедленно и
  никогда не является заходной картой).
- **Оракул бой-стека**: каждая соседняя пара `(Table[i], Table[i+1])` удовлетворяет `CanBeat` — из
  чего автоматически следует **I-7** (над пикой только старшая пика).
- **I-4/I-5** — свойства перехода (не снимка), покрываются golden-сценариями, а не `CheckInvariants`.
- **I-2/I-3** — итерация 4 (ШУХи).
