# View — per-seat проекция состояния (Слой 0)

> **Дата.** 2026-07-16
> **Спец.** Спец 1 (ядро движка, Слой 0), недостающий метод публичного API.
> **Отношение к правилам.** Источник истины — [`shukh-rules.md`](../../shukh-rules.md).
> Модель видимости зафиксирована решением **D-9** в [`architecture.md`](../../architecture.md).
> **Ветка.** `worktree-engine-view`, база — вершина `main` (независимо мержится в `main`;
> параллельна итерации 4).

---

## 1. Цель

Реализовать `View(state, seat)` — последний недостающий метод публичного API Слоя 0
(в §3 архитектуры заявлено `NewGame / LegalActions / Apply / View`). Функция строит
**проекцию состояния для одного места**: отдаёт ровно то, что данный игрок вправе видеть
за реальным столом (D-9), и **структурно исключает утечку скрытых карт** — в типе `View`
для чужих карт просто нет поля.

Не входит: сеть, сериализация, диффы/патчи вида, легальные ходы (остаются в
`LegalActions`). Всё это — Слой 1+ и отдельные функции.

---

## 2. Модель видимости (D-9)

| Сущность | Видимость |
|----------|-----------|
| Своя рука | целиком (карты) |
| Чужая рука | **скрыта**; публично только **число карт** |
| Кон (`Table`) | публичен целиком: карты, порядок, число |
| Отбой (`Discard`) | закрыт (R-2.9): публичен только **размер** стопки |
| ШУХ-карты (`Shukh`) | выданы рубашкой вверх, не входят в счёт руки до взятия (R-6.4, I-3); публичен лишь **счётчик «ожидает»** — свой и чужой |
| Талон | остаток колоды числом (после раздачи = 0, R-4.10) |
| Чей ход / фаза / кто жив / порядок финиша / конфиг партии | публичны |

«Одна карта!» (§6) — ритуал-объявление, не источник скрытой информации; отдельного поля
во `View` не требует.

---

## 3. Форма данных

**Ключевое проектное решение:** скрытые руки представляем **только счётчиками**
(вариант A из брейншторма). Утечь картой невозможно — некуда её положить.

```go
// View — проекция состояния для одного места (seat). Отдаёт только то,
// что этот игрок вправе видеть (D-9). Скрытых карт в типе нет вовсе.
type View struct {
    // публичная конфигурация партии
    Rules RuleSet
    Mode  EnforcementMode
    Phase Phase // Playing | Finished

    You  SeatID // чьё это view
    Turn SeatID // чей сейчас ход (публично)

    // своя рука: целиком
    Hand         []Card // карты на руках (владелец видит)
    ShukhPending int    // сколько своих ШУХ-карт ждёт взятия (I-3)

    // соперники: только счётчики, скрытых карт нет.
    // порядок — по часовой, начиная со следующего за You места (R-2.13).
    Opponents []OpponentView

    // публичный стол
    Table   []TableCard // кон, снизу→вверх (карты и порядок публичны)
    Discard int         // размер закрытого отбоя (содержимое скрыто, R-2.9)
    Talon   int         // остаток колоды (после раздачи = 0, поле общее)

    // итог
    Live   map[SeatID]bool // кто ещё в игре
    Finish []SeatID        // порядок финиша → места (R-9.2)
}

type OpponentView struct {
    Seat         SeatID
    HandCount    int  // число карт в руке (публично)
    ShukhPending int  // число ожидающих ШУХ-карт (публично, I-3)
    Live         bool
}
```

### Решения по форме

1. **`Discard int`** — только размер стопки; содержимое скрыто (R-2.9). За столом высоту
   стопки видно.
2. **`Opponents` в порядке по часовой** начиная со следующего за `You` места — UI сразу
   раскладывает соперников по кругу от «себя» (R-2.13). Сам игрок в список не входит.
3. **`View` — чистая проекция, без легальных ходов.** `LegalActions(state, seat)` остаётся
   отдельной функцией (YAGNI, без дублирования).
4. **`ShukhPending`** — из существующего на `main` поля `Shukh map[SeatID][]Card` (длина
   среза). Поля итерации 4 (`OwesOneCard`, `ShukhTakeable`) здесь **не** проецируются;
   при вливании итерации 4 `View` расширяется аддитивно.

## 4. Поведение / контракт

- `func View(s *State, seat SeatID) View` — чистая функция, **не мутирует** `s`.
- Возвращаемые срезы/карты — **копии** (или новые срезы), чтобы вызывающий не мог
  через `View` дотянуться до внутренних срезов `State` и изменить их. Карты — value-типы,
  копируются тривиально; срезы `Hand`/`Table`/`Finish` и мапа `Live` копируются.
- `seat` обязан быть валидным местом партии (есть в `s.Seats`). Невалидное место — это
  ошибка программиста (не игровое событие), поэтому `View` **паникует**. В плане свериться
  с фактической конвенцией движка на невалидный вход и выровняться под неё, если она иная.
- Детерминизм: одинаковый `(state, seat)` → идентичный `View`. Порядок `Opponents`
  фиксирован (по часовой от `You`), порядок `Table`/`Hand`/`Finish` — как в состоянии.

## 5. Тестирование (TDD)

Тесты в `engine/view_test.go`. Ключевые проверки:

1. **Своя рука видна** — `Hand` совпадает с `state.Hands[seat]` (как множество).
2. **Чужие карты не утекают** — в типе `OpponentView` нет карт; `HandCount` каждого
   соперника равен `len(state.Hands[opp])`.
3. **Порядок соперников** — по часовой начиная со следующего за `You`; сам не входит.
4. **Кон публичен** — `Table` совпадает с `state.Table` по картам и порядку.
5. **Отбой** — `Discard == len(state.Discard)`, карты не отдаются.
6. **ШУХ-счётчики** — `ShukhPending` (свой и чужой) равны `len(state.Shukh[seat])`.
7. **Талон** — `Talon == len(state.Talon)`.
8. **Иммутабельность** — мутация возвращённых срезов не меняет исходный `State`;
   `View` не мутирует вход (сравнить `state` до/после, напр. через `CheckInvariants`
   или глубокое сравнение клона).
9. **Согласованность per-seat** — для каждого места сумма видимого (своя рука) и
   счётчиков соперников согласуется с полным состоянием (число карт по всем местам
   сходится).
10. **Детерминизм** — два вызова с тем же входом дают равные `View`.

Прогон против существующей инфраструктуры: подкладываем точную колоду через `NewGame`
(D-11), проигрываем несколько ходов `Apply`, затем снимаем `View` с каждого места.
