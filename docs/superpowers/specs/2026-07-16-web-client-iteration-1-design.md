# Веб-клиент «Шух» — итерация 1 (каркас) — дизайн

> **Статус.** Согласовано с владельцем 2026-07-16. Это спека первой итерации фронта
> (Слой 3 по [`architecture.md`](../../architecture.md), Спец 3).
>
> **Отношение к движку.** Движок (Спец 1) и его проекция `View(state, seat)` — источник
> формы данных. Сервер (Спец 2, WebSocket + комнаты) ещё не существует, поэтому эта
> итерация строится на **фейковом транспорте** и **фикстурах** и полностью развязана с
> агентами, дорабатывающими движок.

---

## 1. Цель итерации

Заложить каркас браузерного клиента так, чтобы:

- была связная первая итерация фронта: **скелет проекта → контракт с движком → живой рендер стола**;
- фронт был **развязан** с прогрессом по движку/серверу (работает на моке и фикстурах);
- **главный шов** (`Transport`) был спроектирован так, что появление WebSocket-адаптера
  Спеца 2 не потребует переписывать `ui/` и `store/`;
- стол выглядел «сразу красиво» в пределах статической сцены (SVG-карты, дизайн-токены,
  адаптив desktop + mobile), а анимации раздачи/боя были **заложены архитектурно**
  (event-driven переходы), но доводятся позже, когда потечёт реальный поток `Event`.

Итог: `npm run dev` поднимает три экрана (Вход → Лобби → Стол) на фикстуре, без Go-сервера.

---

## 2. Ключевые решения

| # | Решение | Обоснование |
|---|---------|-------------|
| W-1 | **Стек: Vite + React + TypeScript.** | Соответствует D-4 (TS/React). Vite — стандартный быстрый дев-сервер/сборка. |
| W-2 | **Слоистая структура внутри `web/`, повторяющая D-6:** `contract/` → `transport/` → `store/` → `ui/`. | Ядро UI не знает про сеть: транспорт скрыт за интерфейсом. Новый вход (WS) = новая реализация `Transport`, а не переписывание. |
| W-3 | **Ручные TS-типы, зеркалящие `View`/`OpponentView`/`Action`/`Event`/`Card`.** Генерацию из Go — позже. | Движок ещё в движении; авто-генерация дала бы тулинг-оверхед и постоянные ломки. Осознанно держим синхрон вручную (комментом-пометкой). |
| W-4 | **Фейковый `Transport` на фикстуре** (`transport/mock.ts`); `send(action)` пока логирует. | Развязка с несуществующим сервером. Тот же интерфейс потом реализует `ws.ts`. |
| W-5 | **Состояние — Zustand.** | Естественно ложится на будущий realtime-поток `View`/`Event` (push из транспорта в стор). Легче, чем городить это на Context. |
| W-6 | **Стили — CSS Modules + CSS-переменные (дизайн-токены).** Без Tailwind. | «Сразу красиво» без тяжёлого тулинга в каркасе; токены дают единый визуальный язык и задел под тему. |
| W-7 | **Карта — SVG-компонент** (ранг + масть, рубашка для скрытых), без растровых изображений и внешних либ. | Чёткая читаемость спецкарт (Дама♥, 9♦, параметрические `X(Y)`), масштабируемость, ноль внешних зависимостей. |
| W-8 | **Три экрана: Вход → Лобби → Стол** через `react-router`. | Согласованный объём итерации; закладывает навигацию комнаты (D-2) без реального бэка. |

## 3. Структура каталогов

```
web/
  src/
    contract/       # ручные TS-зеркала движка + интерфейс Transport (шов к Спецу 2)
      types.ts      #   Card, View, OpponentView, Action*, Event*
      transport.ts  #   interface Transport { subscribe, send }
    transport/
      mock.ts       # фейковый in-memory Transport поверх фикстуры
    store/
      game.ts       # Zustand: текущий View, лог событий, send(action)
    ui/
      screens/      # Join, Lobby, Table
      table/        # Card, Hand, Con, OpponentSeat, ShukhBadge, ActionBar
      kit/          # Button, Modal и пр. + токены (theme.css)
    fixtures/        # готовые View-снапшоты (стол, лобби)
    App.tsx
    main.tsx
  index.html
  package.json  tsconfig.json  vite.config.ts  .eslintrc  .prettierrc
```

Живёт в `web/` в корне репозитория, рядом с `engine/`. Go и фронт не смешиваются.

## 4. Слой контракта и транспорта (главный шов)

### 4.1 Типы (`contract/types.ts`)

Ручные зеркала Go-типов движка. Каждая группа помечена комментарием
`// зеркало engine/<file>.go — синхронизировать вручную`.

- `Card` — ранг + масть (`'♥' | '♦' | '♣' | '♠'`); параметрические карты несут пометку
  замены (для отрисовки `X(Y)`).
- `View` — поля по `engine/view.go`: `Rules, Mode, Phase, You, Turn, Hand, ShukhPending,
  Opponents, Table, Discard, Talon, Live, Finish`.
- `OpponentView` — `Seat, HandCount, ShukhPending, Live`.
- `Action` (union) — `PlayCard, TakeBottomAndPass, PodkladkaWest, ClaimShukh,
  GiveShukhCard, TakeShukhCards`.
- `Event` (union) — `GameStarted, CardPlayed, ConClosed, ConSwept, PlayerFinished,
  GameFinished, CardsTaken, PodkladkaPlayed, TurnSkipped, ShukhAssessed, ActionReverted,
  ShukhPaid, ShukhCardsTaken`.

### 4.2 Интерфейс `Transport` (`contract/transport.ts`)

```ts
interface Transport {
  subscribe(onView: (v: View) => void, onEvent: (e: Event) => void): () => void
  send(action: Action): void
}
```

### 4.3 Мок (`transport/mock.ts`)

Реализует `Transport` поверх фикстуры: при `subscribe` сразу эмитит стартовый `View`;
`send(action)` пока логирует (позже — скриптованные ответы). **Когда появится сервер
Спеца 2, добавляем `transport/ws.ts` с тем же интерфейсом — `ui/` и `store/` не меняются.**

---

## 5. Стор (`store/game.ts`)

Zustand-стор держит: текущий `View`, лог событий, ссылку на `Transport`. Подписывается на
транспорт (`onView` → обновляет `View`; `onEvent` → добавляет в лог, триггерит переходы).
Экшен `play(action)` → `transport.send(action)`. UI читает только стор — источник данных
(мок/WS) ему безразличен.

---

## 6. Экраны (`ui/screens/`)

- **Join** — имя + код комнаты, кнопка «Войти» → `navigate('/room/:code')`. Бэка нет, код
  валидируется формально.
- **Lobby** — список игроков (из фикстуры), индикатор готовности, «Начать» →
  `/room/:code/table`.
- **Table** — центральная сцена из стора (питается моком):
  - кон в центре (`Table` из View, порядок карт);
  - своя рука снизу (веер, `Hand`);
  - соперники по кругу (`OpponentSeat`: имя, число карт, бейдж `ShukhPending` — ожидающие
    ШУХ-карты, публичен лишь счётчик, I-3);
  - `ActionBar` — кнопки-заглушки «ШУХ!», «Одна карта!», «Взять низ». Легальность действий
    подсвечивается по `View`; полноценный `LegalActions` придёт с сервером.

---

## 7. Визуал / дизайн-система (`ui/kit/theme.css`)

- **Токены** (CSS-переменные): палитра сукна, красный/чёрный мастей, радиусы, тени,
  типографика. Переменные заложены под светлую/тёмную тему (сама тема — по желанию позже).
- **Карта — SVG-компонент**: ранг + `♥♦♣♠`, рубашка для скрытых. Спецкарты читаются:
  Дама♥, 9♦; параметрические `6(2)♥` рисуются как «6» с пометкой замены.
- **Раскладка адаптивная** (desktop + mobile): стол — CSS grid; веер руки — flex; позиции
  соперников — по кругу.
- **Анимации**: заложить **event-driven переходы** (карта уходит в кон; кон уезжает в
  отбой) на CSS-transition. Сейчас реально работают: ховер, выбор карты, лёгкий переход
  хода. «Богатая» раздача/бой — позже, когда потечёт реальный `Event`-поток.

---

## 8. Тулинг и проверка

- ESLint + Prettier + `tsconfig` strict.
- Smoke-тесты (**Vitest + React Testing Library**):
  - мок отдаёт `View` → `Table` рендерит нужное число карт руки и мест соперников;
  - клик по карте зовёт `transport.send(PlayCard)`.
- `npm run dev` — живой стол на фикстуре в браузере без Go-сервера.

---

## 9. Явно вне этой итерации (YAGNI)

Реальный WebSocket и реконнект-токен; генерация типов из Go; голосование/оспаривание
(R-8.6); реальный `LegalActions` с сервера; полноценные анимации раздачи/боя; аккаунты и
история; серия партий (R-10.2).

---

## 10. Как это стыкуется с дорожной картой

Это первый кусок **Спеца 3** (Слой 3), сознательно начатый до завершения Спеца 2. Шов
`Transport` (W-2/W-4) — точка стыковки: когда Спец 2 даст WebSocket-адаптер, фронт получает
`transport/ws.ts` вместо мока, и остальной клиент не трогается. Форма данных фиксируется по
`View`/`Action`/`Event` движка (Спец 1), синхрон держим вручную (W-3) до появления
генерации типов.
