## 1. Цель и рамки

**Цель.** Транспорт-агностичная **оркестрация одной живой партии** поверх движка (Слой 1,
D-6): авторитетное состояние, идентичность игроков, жизненный цикл матча, приём действий,
per-seat проекция вида и поток событий. Плюс — достройка отложенной из Спеца 1
машинерии **голосования/оспаривания** (R-8.6) и субъективных ШУХов в самом движке.

Итог — сетевой-агностичный «сервис партии», гоняемый тестами в памяти (как движок), поверх
которого Слой 2 (WS + комнаты) встанет тонким адаптером без переписывания.

**В рамках:**
- **A1. Достройка движка** — `State.Pending *Adjudication`, действия `ClaimSubjective`,
  `Dispute` и `Vote`, подсчёт кворума и применение исхода через существующую §8-машинерию.
  Субъективные ШУХи Ш-6/Ш-9/Ш-10 и оспаривание→Ш-8 (R-8.6, R-8.9). Всё чистое и golden-тестируемое.
- **A2. `game.Session`** — идентичность игроков (`PlayerID↔SeatID`), жизненный цикл
  `Lobby→Playing→Finished`, синхронный приём действий, проекция `SeatView`+`LegalActions`,
  фанаут событий подписчикам.

**Вне рамок (следующие спеки / отложено):**
- **Слой 2** — WebSocket-сервер, комнаты по коду/ссылке, реконнект-токены (D-2). Отдельный спек.
- **Тайм-ауты хода (OQ-2)** — Session синхронна и часов не держит. Реальное время и таймеры
  живут в транспорте; решается в Слое 2. Субъективный Ш-6 «завис» ловится вручную (голосование).
- **Режим Culture (D-10)** — как и в Спеце 1: enum присутствует, реализация отложена.
- **Серия партий (R-10.2)**, аккаунты/история — «позже» по дорожной карте.

**Критерий-девиз.** Session — тонкий авторитетный оркестратор: **правила живут в движке**
(D-1/D-6), Session лишь держит состояние, сопоставляет игроков сиденьям, проводит процедуры
и проецирует видимость. Любой инвариант §11 держится после каждого применённого действия.

## 3. Достройка движка — Adjudication / голосование (A1)

Отложено из Спеца 1 (engine-core-design, стр. 392–394: «В Слое 0 голосование — плумбинг без
потребителя»). В Спеце 2 потребитель есть (мультиплеер), поэтому достраиваем — **в движке**,
следуя исходной модели (`Pending *Adjudication`, действие `Vote`).

**Модель по R-8.6 — оспаривание, не симметричное голосование.** Предъявленный субъективный
ШУХ по умолчанию **стоит на цели**. Оспорить его может любой (обычно — цель); голосование —
на стороне оспаривающего. Только если оспаривающего поддержало **большинство стола** — ШУХ
**перекладывается на предъявителя** (Ш-8, R-8.6). Недобор/ничья → оспаривание не прошло → ШУХ
остаётся на цели. Это снимает неоднозначность исхода на границе.

### 3.1 Состояние
```go
type Adjudication struct {
    Claimant SeatID              // кто предъявил ШУХ (P-1: живое сиденье, не Target)
    Target   SeatID              // на ком ШУХ стоит по умолчанию
    Code     ShukhCode           // повод (субъективный: Ш-6/Ш-9/Ш-10)
    Disputer SeatID              // кто оспаривает (открыл разбор)
    Votes    map[SeatID]bool     // голоса: true = поддержать оспаривающего
}
// в State:
//   Pending *Adjudication       // активный разбор (R-8.6), nil если нет
```
**Гейт-инвариант:** пока `Pending != nil`, стол «замер на разборе» — легальны только `Vote` и
`Resolve` (закрытие разбора); обычные ходы отвергаются до `VoteResolved`.

### 3.2 Действия
- `ClaimSubjective{Target, Code}` — предъявить субъективный ШУХ (Ш-6 «завис»/Ш-9 «зря
  крикнул»/Ш-10 «небрежность»). Actor-agnostic (P-1): предъявляет любое живое сиденье, кроме
  `Target`. Сам по себе — ставит ШУХ на `Target` по §8-машинерии (как авто-ШУХ).
- `Dispute{}` — цель (или иной игрок) открывает оспаривание → `Pending`, эмитит `VoteOpened`.
- `Vote{Support bool}` — голос за/против оспаривающего. **Кворум — большинство стола, включая
  вышедших** (R-8.9): вышедшие продолжают судить (R-9.5). Голос применяется как подавшее сиденье.

### 3.3 Исход (`VoteResolved`)
- **Большинство поддержало оспаривающего** → Ш-8 «ложный ШУХ»: штраф снимается с `Target` и
  перекладывается на `Claimant` через **ту же §8 payment-машинерию** (payment-гейт, зона ШУХ,
  откат при необходимости).
- **Иначе (недобор/ничья)** → ШУХ остаётся на `Target`. `Pending` очищается, стол «оживает».
- Движок **не оценивает субъективную суть** — только считает голоса и применяет мех. исход.

### 3.4 События
Добавляются `VoteOpened{Claimant, Target, Code, Disputer}` и `VoteResolved{Code, Upheld bool}`
(`Upheld` = оспаривание удалось, ШУХ ушёл на `Claimant`), рядом с существующими
`ShukhAssessed`, `ActionReverted`, `ShukhPaid`.

### 3.5 Режимы и тесты
- Guard/Middle не меняются; субъективные ШУХи доступны во всех режимах (D-10 — субъективные
  всегда предъявление+голосование).
- **Тесты:** golden-путь на каждый субъективный ШУХ (Ш-6/8/9/10) и на каждый исход
  голосования (подтверждение, Ш-8-переклад, недобор); гейт обычных ходов при `Pending`;
  инварианты §11 держатся после разбора.

## 4. `game.Session` — идентичность, жизненный цикл, API (A2)

### 4.1 Идентичность и ростер
- `PlayerID` — стабильная непрозрачная строка (выдаётся вызывающим Слоем 2; Session её лишь
  хранит и сопоставляет). `Name` — отображаемое имя.
- Session держит `PlayerID↔SeatID` и знает хоста (создателя). Про сокеты/токены/коды — не знает (S-5).

### 4.2 Жизненный цикл `Lobby → Playing → Finished`
- **Lobby.** Session создаётся с `Config` (`RuleSet` + `EnforcementMode` + размер колоды,
  D-8/D-10) и хостом. Игроки входят (`Join`) до 2–8 (D-3). Порядок входа фиксирует рассадку по
  часовой (R-2.13). Хост может менять `Config` и стартовать.
- **Start.** Хост зовёт `Start(host, seed)` → колода тасуется `shuffle` по seed (D-11) →
  `engine.NewGame(cfg, deck)` → `Phase→Playing`. (В тестах — инъекция точной колоды, S-7.)
- **Finished.** Движок дошёл до `Finished` (R-9.2) → Session фиксирует места (`Finish`).

### 4.3 API (синхронный, под `sync.Mutex`, S-3)
```go
func NewSession(cfg Config, host PlayerID) *Session

func (s *Session) Join(playerID PlayerID, name string) error   // только в Lobby, до N
func (s *Session) Leave(playerID PlayerID)
func (s *Session) SetConfig(host PlayerID, cfg Config) error    // только хост, только в Lobby
func (s *Session) Start(host PlayerID, seed int64) error        // только хост, Lobby→Playing

func (s *Session) Submit(playerID PlayerID, a engine.Action) ([]engine.Event, error)
func (s *Session) Subscribe(playerID PlayerID) (<-chan Update, func()) // func() = unsubscribe
func (s *Session) Snapshot(playerID PlayerID) Update
```
- `Submit` проверяет `playerID`→сиденье и право (его очередь для turn-actions; право сиденья
  для actor-agnostic действий — ClaimShukh/Vote/AskCount, как в движке, P-1), затем зовёт
  `engine.Apply`, обновляет держимое `State`, возвращает события подавшему и фанаутит остальным (§5).
- Действия — те же типы `engine.Action` (включая новые `ClaimSubjective`/`Vote` из §3).

## 5. Проекция и доставка событий

- **Единица доставки** повторяет уже существующий веб-контракт (S-6):
  ```go
  type Update struct {
      View   engine.SeatView   // per-seat проекция (D-9): своя рука видна, чужие — счётчики
      Legal  []engine.Action   // зеркало engine.LegalActions для этого сиденья
      Events []engine.Event    // новые события с прошлого Update
  }
  ```
  После каждой мутации Session пере-проецирует `View`/`Legal` для каждого подписчика и шлёт
  `Update`. Веб (итер. 1/2a) потребляет ровно `GameSnapshot{view, legal}` + поток `GameEvent`.
- **Slow-consumer policy.** Каналы подписчиков буферизованы; отправка неблокирующая. При
  переполнении — дельта дропается, подписчик помечается «пересними снапшот» (клиент перечитает
  `Snapshot(playerID)`). Игра под mutex не блокируется медленным потребителем.
- **Защита.** `engine.CheckInvariants(state)` после каждого `Apply` — «железность» §11.

## 6. Обработка ошибок

- **Нелегальное действие** → типизированная `engine.IllegalAction` возвращается **только
  подавшему**; состояние не тронуто, событий нет, фанаута нет.
- **Session-уровневый отказ** (не твоё сиденье, не твоё право, действие вне текущей фазы
  Lobby/Playing, `Join` после старта или сверх N) → отдельная типизированная ошибка `game.*`,
  тоже без мутации и фанаута.
- **Никаких паник** в штатных путях. Нарушение инварианта §11 после `Apply` — это баг движка,
  его не маскируем (в dev/тестах — паника/fail, не «тихая» порча).

## 7. Тесты и границы

- **Движок (A1):** golden на каждый субъективный ШУХ (Ш-6/8/9/10) и исход голосования; гейт
  обычных ходов при `Pending`; инварианты §11 после разбора.
- **Session (A2):** in-memory сквозняк `join → start → разыграть скриптованную партию →
  предъявление субъективного ШУХа + голосование → finish`; параллельные `Submit` под mutex
  (гонки — `go test -race`); проекция скрывает руки соперников (D-9); ошибки §6 не мутируют
  состояние и не фанаутят; slow-consumer не блокирует игру. Стиль — как в `engine/`
  (детерминизм, точная колода, S-7).
- **Явно вне тестов этого спека:** сеть/WS, реконнект-токены, тайм-ауты (Слой 2 / OQ-2).

## 9. Как это стыкуется с дорожной картой

- Закрывает **Слой 1** Спеца 2 (`game/ session`, D-6) и достраивает отложенную из Спеца 1
  голосовательную машинерию в движке.
- Оставляет **честный шов** для Слоя 2: WS-сервер + комнаты (D-2) отобразят соединение →
  `PlayerID` и код комнаты → `Session`, дергая её синхронный API и ретранслируя `Update`.
  UI-контракт (`SeatView`+`legal`+события) уже совпадает с веб-клиентом — адаптер без правок UI.
- Открытые вопросы, адресуемые Слоем 2: **OQ-2** (тайм-ауты хода), **OQ-3** (UI/кворум
  голосования онлайн — механика кворума здесь уже в движке, остаётся сетевой UX).
