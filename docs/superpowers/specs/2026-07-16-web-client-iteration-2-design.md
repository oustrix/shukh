# Веб-клиент «Шух» — итерация 2 (живой + играбельный стол + ритуал ШУХа) — дизайн

> **Статус.** Согласовано с владельцем 2026-07-16. Продолжение Слоя 3
> ([`architecture.md`](../../architecture.md), Спец 3), поверх каркаса итерации 1.
>
> **База.** Ветвится от `main` (`b8613c3` — итерация 1 веба уже влита). Движок на
> main предоставляет реальные `engine/view.go` (`SeatView`) и `engine/legal.go`
> (`LegalActions`) — контракт сверяем с ними.
>
> **Отношение к серверу.** Спец 2 (game-session + WebSocket + комнаты, Слои 1–2)
> ещё не существует. Эта итерация углубляет **клиент** на **скриптованном моке**,
> который остаётся честным стендом под будущий `ws.ts`: клиент — зритель пушей, а
> правила живут в Go (движок/сервер), не в клиенте.

---

## 1. Цель итерации

Оживить и «оиграбелить» стол, не уходя от мока и не дублируя правила:

- **Живой стол** — карты движутся между зонами (раздача, заход/бой, взятие низа,
  закрытие+смётание кона, уход карты в ШУХ-зону) через анимации, привязанные к
  потоку событий.
- **Играбельный стол** — интерактив: подсветка легальных ходов (из снапшота),
  выбор+подтверждение карты, корректные состояния кнопок; действие продвигает
  скриптованный таймлайн.
- **Ритуал ШУХа** — кнопки «ШУХ!»/«Одна карта!», отображение ШУХ-зон и штрафов,
  модалка голосования/оспаривания (R-8.6, голоса скриптованы).

Итог: `npm run dev` из `web/` проигрывает связный **скриптованный сценарий** партии —
стол оживает и реагирует на игрока — без Go-сервера.

## 2. Ключевые решения

| # | Решение | Обоснование |
|---|---------|-------------|
| W2-1 | **Скриптованный транспорт вместо emit-once мока.** `Scenario` = список шагов; `auto`-шаги играются по таймеру, `await`-шаги ждут ожидаемого действия игрока и продвигают таймлайн. | Даёт «живость» и интерактив без правил в клиенте. Тот же интерфейс `Transport` — `ws.ts` Спеца 2 встанет без правок UI (честный стенд). |
| W2-2 | **Легальность приходит в снапшоте (`legal: Action[]`), клиент её не считает.** Зеркало `LegalActions(s,seat)`, которое сервер шлёт рядом с `SeatView`. | Правила — источник истины в Go. Клиент только подсвечивает. Не плодим расхождения (ср. W-3). |
| W2-3 | **Контракт сверяется с реальным движком:** `View` → `SeatView` (поля из `engine/view.go`), снять устаревший комментарий «view.go не реализован». | view.go влит в main; мирроринг теперь имеет с чем сверяться. |
| W2-4 | **Стабильная идентичность карты** (`suit+rank`, карты в колоде уникальны) как React-ключ/`layoutId` вместо `key={i}`. | Несущее для FLIP-анимаций (motion трекает элемент по ключу между зонами). Ранее — iteration-2 cleanup из ревью, теперь обязательное. |
| W2-5 | **Анимации на `motion` (framer-motion).** Единственная новая зависимость. | `layout`/`layoutId` даёт FLIP-полёт карт автоматически; `AnimatePresence` — вход/выход. Ручной FLIP для карточной игры — много хрупкого кода. |
| W2-6 | **Свободная игра по правилам — вне итерации.** Мок реагирует только на ожидаемый ход сценария; офф-скрипт мягко блокируется. | Настоящая реакция на произвольный ход требует правил (Go→WASM/дублирование в TS) — большой лифт, отложено до Спеца 2. |
| W2-7 | **Голосование R-8.6 — скриптованное.** Голоса ботов заданы сценарием; кворум-логику по-настоящему не считаем. | Это UI-поток; настоящий подсчёт голосов — забота сервера. |
| W2-8 | **Уважаем `prefers-reduced-motion`** — при нём переходы мгновенные. | Доступность; тесты гоняются в reduced-режиме (детерминизм). |

## 3. Контракт и мок-драйвер (Секция 1, несущая)

### 3.1 Контракт (`src/contract/`)
- `View` → **`SeatView`**, поля сверены с `engine/view.go`
  (`Rules, Mode, Phase, You, Turn, Hand, ShukhPending, Opponents, Table, Discard,
  Talon, Live, Finish`). Комментарий-зеркало обновить на `engine/view.go` без
  оговорки «не реализовано».
- `GameSnapshot = { roomCode; seats: SeatMeta[]; view: SeatView | null; legal: Action[] }`
  — `legal` = легальные ходы текущего игрока (зеркало `LegalActions`, W2-2).
- Хелпер `cardKey(card): string` (`suit+rank`) — стабильный ключ (W2-4).
- Хелпер `isLegal(legal, action): boolean` для подсветки.

### 3.2 Скриптованный транспорт (`src/transport/scripted.ts`)
```ts
interface Step {
  kind: 'auto' | 'await'
  expect?: Action          // для 'await': ожидаемое действие игрока
  events: GameEvent[]      // эмитятся при входе в шаг
  snapshot: GameSnapshot   // состояние после шага
  delayMs?: number         // для 'auto': пауза перед проигрыванием
}
type Scenario = Step[]
function createScriptedTransport(scenario: Scenario): Transport
```
- `subscribe`: играет шаги. `auto` — по таймеру (`delayMs`): эмитит `events`, затем
  `snapshot`, идёт к следующему. `await` — останавливается.
- `send(action)`: на `await`-шаге сверяет с `expect`; совпал — проигрывает события и
  снапшот шага, продолжает до следующего `await`; не совпал — мягко игнор (UI покажет
  «не сейчас», но правил клиент не判ит).
- Таймеры детерминированно управляемы в тестах (инъекция «тика»/фейк-таймеры).

### 3.3 Стор
- `events`-лог получает лёгкий предел длины (кольцевой буфер, последние N) — закрывает
  iteration-2 minor из ревью (иначе O(N²) рост).

## 4. Живой стол — анимации (Секция 2)

- Зоны (рука, кон, отбой-счётчик, ШУХ-зоны) — `motion`-контейнеры; карта несёт
  `layoutId = cardKey`. Переход карты между массивами (`hand`→`table` и т.п.) motion
  анимирует как FLIP-полёт.
- `AnimatePresence` на вход/выход: раздача (влёт в руки), смётание кона (уезжает+fade),
  уход карты в ШУХ-зону по `ShukhPaid`.
- Событийные акценты: вспышка при `ShukhAssessed`, подсветка закрытия кона
  (`ConClosed`→`ConSwept`), «взять низ» (`CardsTaken`).
- `prefers-reduced-motion` → переходы отключены (W2-8).

## 5. Играбельный стол — интерактив (Секция 3)

- **Подсветка легальных ходов** из `snapshot.legal`: нелегальные карты руки притушены
  и некликабельны; `ActionBar` включает кнопки по легальности. Клиент правил не считает.
- **Выбор + подтверждение:** клик по легальной карте → выбор (приподнимается);
  повторный клик / кнопка «Сходить» → `play({type:'playCard', card})`. Не свой ход →
  рука неинтерактивна.
- Действие → `store.play` → `scripted.send` сверяет с `expect` → продвигает таймлайн
  (события+снапшот → секция 4 анимирует диф).
- **A11y:** `onKeyDown` (Enter/Space) на карте (iteration-2 cleanup из ревью),
  видимый фокус.

## 6. Ритуал ШУХа (Секция 4)

- **ШУХ-зоны:** у каждого места отложенная стопка (рубашкой, только счётчик, I-3);
  своя — когда `ShukhTakeable`. Клик по своей (после кона) → `takeShukhCards`.
- **«ШУХ!»**: скриптованный шаг открывает catch-window (напр. бот зашёл с Дамы♥);
  игрок жмёт «ШУХ!» → `claimShukh` → `ShukhAssessed`, карта уезжает в ШУХ-зону
  нарушителя (анимация §4).
- **«Одна карта!»**: кнопка-объявление (сбрасывает `OwesOneCard`); на моке —
  скриптованное событие. Забыл вовремя → скриптованный Ш-11 ловится «ШУХ!».
- **Голосование/оспаривание (R-8.6):** субъективный ШУХ поднимает модалку; голоса
  ботов скриптованы (W2-7); показываем исход и применяем штраф. Кворум по-настоящему
  не считаем.

## 7. Тесты и границы (Секция 5)

- **Компонентные тесты:** подсветка по `legal` (какие карты интерактивны);
  выбор+подтверждение зовёт `play`; «ШУХ!» открывает модалку; подсчёт голосов
  отображается; A11y-активация карты с клавиатуры.
- **Интеграционный тест сценария:** прогон `createScriptedTransport` по сценарию
  (детерминированные таймеры) — стор проходит ожидаемые снапшоты, UI отражает ключевые
  состояния (без проверки кадров анимации).
- **Анимации не юнит-тестим** (визуально). Прикрываем `prefers-reduced-motion` +
  смоук, что motion-компоненты рендерятся. Гейт остаётся `typecheck && lint && test`.
- **Явно вне итерации (YAGNI):** реальный `ws.ts`; Go→WASM / свободная игра по
  правилам (W2-6); реконнект; аккаунты; настоящая кворум-логика голосования (W2-7);
  тайм-ауты хода (OQ-2).

---

## 8. Как это стыкуется с дорожной картой

По-прежнему Спец 3, но клиент теперь готов принимать **поток** снапшотов/событий и
легальность — ровно то, что даст `ws.ts` Спеца 2. Скриптованный транспорт (W2-1) —
двойник сервера: когда появится Спец 2, `transport/ws.ts` реализует тот же `Transport`,
и `ui/`/`store/` не меняются. Контракт синхронизирован с реальным `SeatView`/
`LegalActions` (W2-2/W2-3), синхрон держим вручную до генерации типов.
