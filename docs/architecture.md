---

## 1. Цель

Дать возможность играть в «Шух» с друзьями **в браузере** (десктоп + мобильный), через
комнату по коду/ссылке. Игра — сетевая, состояние авторитетно на сервере (карты
соперников скрыты — клиенту доверять нельзя).

## 2. Ключевые решения (с обоснованием)

| D-2 | **Точки входа — комната по коду/ссылке, без аккаунтов** (в MVP). Ввёл имя — играешь. Реконнект по токену в ссылке. | Быстрый путь к «поиграть с друзьями». Аккаунты/история — возможное будущее (см. дорожную карту). |
| D-3 | **Игроков 2–8.** Учитываем эндшпиль на 2 игрока (§9.2: сброс `6(2)♥`). | Диапазон от дуэли до большой компании. |
| D-4 | **Стек: Go (движок + сервер) + WebSocket + TypeScript/React (фронт).** | Путь проекта — Go; движку нужен авторитетный сервер; карточный UI с перетаскиванием проще на React. Реалтайм — WebSocket. |
| D-5 | **Колода 36 и 52, по умолчанию 36.** Параметрические карты `X(Y)` по R-2.3. | Правила уже параметрические; поддержать обе дёшево. |
| D-6 | **Строгая слоёная архитектура: ядро не знает про сеть и UI** (см. §3). | Закладывает будущие точки входа (Telegram-бот/mini-app, нативка) как тонкие адаптеры — без переписывания ядра. |
| D-7 | **Движок детерминирован и сидируем** (RNG передаётся снаружи; тасовка воспроизводима по seed). | Воспроизводимые партии → тестируемость (в т.ч. против инвариантов §11), отладка, реплеи. |

## 3. Слоёная архитектура

Ключевая идея — **ядро ничего не знает ни про сеть, ни про UI**. Тогда новые точки
входа — это адаптеры поверх одного и того же ядра, а не переписывание.

```
Слой 0  engine/         чистый Go, детерминированный, БЕЗ I/O и сети
        └─ вся логика правил: колода(36/52), раздача §4, бой §3, коны §5,
           ШУХи §7–§8, эндшпиль §9, инварианты §11. RuleSet с флагами вариантов §12.
           API: NewGame / LegalActions(state,seat) / Apply(state,action) / View(state,seat).

Слой 1  game/ (session)  оркестрация ОДНОЙ живой партии поверх engine:
        └─ авторитетное состояние, приём действий, проекция «вида» на игрока,
           поток событий. ТРАНСПОРТ-АГНОСТИЧНЫЙ интерфейс. ← главный шов для точек входа.

Слой 2  адаптеры/входы   Сейчас: WebSocket-сервер + комнаты (код/ссылка, реконнект-токен).
        └─ Позже: Telegram-бот, Telegram mini-app, нативка — новые адаптеры над Слоем 1.

Слой 3  web/ (React/TS)  браузерный клиент: стол, рука, тапы/drag, кнопки «ШУХ!»,
        └─ «Одна карта!», голосование. Десктоп + мобильный.
```

**Как это закрывает требования:**

- **Множество точек входа в будущем** → Слои 0 и 1 не содержат кода про WebSocket/HTTP/
  Telegram. Новый вход = тонкий адаптер над `game.Session`. Браузер и Telegram делят один
  и тот же код игры.
- **Варианты §12** → `RuleSet` (структура флагов) читается правилами движка; сейчас флаги
  выключены, но ветвления размечены.
- **Гибрид-энфорсмент** → `LegalActions(state, seat)` позволяет UI гасить нелегальные ходы,
  а серверу — отвергать жульё. Субъективные ШУХи/голосование — отдельные типы действий,
  которые движок проводит по процедуре, не «судя» по сути.

## 4. Дорожная карта спеков

Стратегия — **движок сначала** (engine-first), разбитый на несколько спеков/итераций.
Каждый спец — свой цикл спек → план → реализация.

1. **Спец 1 — ядро движка (Слой 0).** Чистый Go, детерминированный, полностью покрыт
   тестами против инвариантов §11. Возможные внутренние итерации: карты+бой §3 → раздача §4
   → жизненный цикл кона §5 → ШУХи §7–§8 + эндшпиль §9. Результат — безголовая игра,
   которую гоняем тестами/CLI-харнессом. *(Сейчас в детальном проектировании.)*
2. **Спец 2 — game-session + WebSocket-сервер + комнаты (Слои 1–2).** Транспорт-агностичный
   сервис партии + WS-адаптер, коды комнат, реконнект.
3. **Спец 3 — React-фронт (Слой 3).** Отзывчивый стол, взаимодействия, кнопки
   ШУХ/голосование/«Одна карта».
4. **Позже.** Telegram-адаптер, опциональные варианты §12, аккаунты/история, серия партий
   (R-10.2), матч-статистика.

## 5. Открытые вопросы (решить при детальном проектировании соответствующего спека)

- **OQ-1. Модель видимости руки.** R-2.7/R-4.2 описывают стопку «лицом вверх», но §6
  («Одна карта!» / «Сколько карт?») со штрафом Ш-11 подразумевает, что число карт соперника
  **не** очевидно автоматически. Нужно с владельцем зафиксировать: что именно видят соперники
  в фазе игры (только верхнюю карту стопки? число карт? ничего?). Движок проектируем с
  **per-seat проекцией `View`**, чтобы поддержать любую трактовку. → уточнить в Спеце 1.
- **OQ-2. Тайм-ауты хода.** Нужны ли ограничения по времени на ход в онлайне (в оффлайне их
  нет; есть субъективный Ш-6 «затянул ход»). → решить в Спеце 2.
- **OQ-3. Процедура голосования/оспаривания** (R-8.6, большинство стола) в онлайне: UI,
  кворум, роль вышедших игроков (R-8.9, R-9.5). → детализировать в Спецах 1–2.
