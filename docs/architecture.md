---

## 1. Цель

Дать возможность играть в «Шух» с друзьями **в браузере** (десктоп + мобильный), через
комнату по коду/ссылке. Игра — сетевая, состояние авторитетно на сервере (карты
соперников скрыты — клиенту доверять нельзя).

## 2. Ключевые решения (с обоснованием)

| # | Решение | Обоснование |
|---|---------|-------------|
| D-1 | **Гибридный энфорсмент правил.** Движок блокирует грубые/формализуемые нарушения (ход не в свою очередь, нелегальный бой по мастям §3, счёт закрытия кона R-5.5, автоматизация раздачи §4) и проводит процедуры. Субъективные ШУХи («замешкался» Ш-6, «небрежность» Ш-10) и социальные объявления («Одна карта!» §6) — через кнопки игроков + голосование/оспаривание (R-8.6). | Живость настольной игры сохраняется, но снимается механическая рутина. Субъективные штрафы по природе не формализуемы (см. предупреждение в §7 правил). |
| D-2 | **Точки входа — комната по коду/ссылке, без аккаунтов** (в MVP). Ввёл имя — играешь. **Реконнект по HttpOnly-куке** (ревизия 2026-07-17, L2-6): инвайт-ссылка `/r/CODE` несёт только код (её шарят), личный токен места выдаётся HTTP-шагом `join` и живёт в HttpOnly-куке пути комнаты. | Быстрый путь к «поиграть с друзьями». Шаримая ссылка не должна содержать личное право на место; HttpOnly-токен не читаем из JS и не попадает в URL/логи. Аккаунты/история — возможное будущее. |
| D-3 | **Игроков 2–8.** Учитываем эндшпиль на 2 игрока (§9.2: сброс `6(2)♥`). | Диапазон от дуэли до большой компании. |
| D-4 | **Стек: Go (движок + сервер) + WebSocket + TypeScript/React (фронт).** | Путь проекта — Go; движку нужен авторитетный сервер; карточный UI с перетаскиванием проще на React. Реалтайм — WebSocket. |
| D-5 | **Колода 36 и 52, по умолчанию 36.** Параметрические карты `X(Y)` по R-2.3. | Правила уже параметрические; поддержать обе дёшево. |
| D-6 | **Строгая слоёная архитектура: ядро не знает про сеть и UI** (см. §3). | Закладывает будущие точки входа (Telegram-бот/mini-app, нативка) как тонкие адаптеры — без переписывания ядра. |
| D-7 | **Движок детерминирован и сидируем** (RNG передаётся снаружи; тасовка воспроизводима по seed). | Воспроизводимые партии → тестируемость (в т.ч. против инвариантов §11), отладка, реплеи. |
| D-8 | **`RuleSet` с фичефлагами с первого дня.** Опциональные варианты §12 (общая «подкладка снизу» V-1…V-4, джокеры) **выключены** в MVP, но точки ветвления в движке размечены. | Включить варианты потом = не переписывать движок (требование владельца). |
| D-10 | **Строгость энфорсмента — режим игры (3 уровня), дефолт «Середина».** Настраивается на комнату (часть конфига партии, рядом с `RuleSet`). **Guard** (макс. защиты): движок предотвращает всё детектируемое. **Middle** (дефолт): блокирует лишь ломающее целостность (ход не в очередь Ш-7, нелегальный бой §3/I-7, карты нет в руке), а social-gotcha (заход с Дамы♥ Ш-2, ранний забор ШУХа Ш-3, тайминг отбоя Ш-4/Ш-5, забыл «одна карта» Ш-11) — **разрешает** и даёт ловить кнопкой «ШУХ!». **Culture** (макс. культуры ШУХа): блокирует лишь физически невозможное; даже ход не в очередь/нелегальный бой ловятся вручную. Субъективные ШУХи (Ш-6, Ш-8, Ш-9, Ш-10) во всех режимах — предъявление+голосование. | Соль «Шуха» — в ловле ошибок (R-1.4); одна жёсткая политика её убивает. Режим отдаёт баланс столу. **Оговорка:** Culture требует, чтобы движок допускал «неправильную» позицию и потом откатывал её при поимке (R-1.4/R-8.5) — это машинерия отката состояния, поэтому Спец 1 реализует Guard+Middle, а Culture размечается в enum, но выносится в позднюю итерацию. |
| D-11 | **Тасовка — на границе ядра.** `engine.NewGame(cfg, deck []Card)` принимает **уже упорядоченную** колоду; сид-тасовка (Fisher–Yates на `math/rand/v2`) живёт в отдельном пакете `shuffle`, который импортирует `engine` (не наоборот). | Сохраняет чистоту Слоя 0 (`engine` без `math/rand`/I/O — следствие D-6/D-7); детерминизм задаётся на границе по `seed` (D-7). Даёт тривиальные golden-тесты раздачи §4: тест подкладывает **точную** колоду вместо перебора сидов под нужный расклад. |
| D-9 | **Модель видимости (`View`): скрытые руки, число карт видно.** Свою руку игрок видит целиком; у соперников содержимое руки **скрыто**, но **количество карт публично**. Кон (`table`) публичен (карты, порядок, число). Отбой закрыт (R-2.9). ШУХ-карты выданы рубашкой вверх и не входят в счёт руки до взятия (R-6.4, I-3) — публичен лишь их счётчик «ожидает». «Одна карта!» (§6) — ритуал-объявление, не источник скрытой инфы. Раздача §4 автоматизирована движком, поэтому «публичность верхней карты стопки при раздаче» — не забота клиента. | Согласовано с владельцем как реальный способ игры за столом. Даёт чистую per-seat проекцию: движок отдаёт каждому его карты + счётчики чужих. Разрешает OQ-1. |

## 3. Слоёная архитектура

Ключевая идея — **ядро ничего не знает ни про сеть, ни про UI**. Тогда новые точки
входа — это адаптеры поверх одного и того же ядра, а не переписывание.

```
Слой 0  engine/         чистый Go, детерминированный, БЕЗ I/O и сети
        └─ вся логика правил: колода(36/52), раздача §4, бой §3, коны §5,
           ШУХи §7–§8, эндшпиль §9, инварианты §11. RuleSet с флагами вариантов §12.
           API: NewGame / LegalActions(state,seat) / Apply(state,action) / View(state,seat).

Слой 1  game/ (session)  оркестрация ОДНОЙ живой партии поверх engine:
        └─ авторитетное состояние, приём действий, проекция «вида» на игрока,
           поток событий. ТРАНСПОРТ-АГНОСТИЧНЫЙ интерфейс. ← главный шов для точек входа.

Слой 2  адаптеры/входы   Сейчас: WebSocket-сервер + комнаты (код/ссылка, реконнект-токен).
        └─ Позже: Telegram-бот, Telegram mini-app, нативка — новые адаптеры над Слоем 1.

Слой 3  web/ (React/TS)  браузерный клиент: стол, рука, тапы/drag, кнопки «ШУХ!»,
        └─ «Одна карта!», голосование. Десктоп + мобильный.
```

**Как это закрывает требования:**

- **Множество точек входа в будущем** → Слои 0 и 1 не содержат кода про WebSocket/HTTP/
  Telegram. Новый вход = тонкий адаптер над `game.Session`. Браузер и Telegram делят один
  и тот же код игры.
- **Варианты §12** → `RuleSet` (структура флагов) читается правилами движка; сейчас флаги
  выключены, но ветвления размечены.
- **Гибрид-энфорсмент** → `LegalActions(state, seat)` позволяет UI гасить нелегальные ходы,
  а серверу — отвергать жульё. Субъективные ШУХи/голосование — отдельные типы действий,
  которые движок проводит по процедуре, не «судя» по сути.

## 4. Дорожная карта спеков

Стратегия — **движок сначала** (engine-first), разбитый на несколько спеков/итераций.
Каждый спец — свой цикл спек → план → реализация.

1. **Спец 1 — ядро движка (Слой 0).** Чистый Go, детерминированный, полностью покрыт
   тестами против инвариантов §11. Возможные внутренние итерации: карты+бой §3 → раздача §4
   → жизненный цикл кона §5 → ШУХи §7–§8 + эндшпиль §9. Результат — безголовая игра,
   которую гоняем тестами/CLI-харнессом. *(Сейчас в детальном проектировании.)*
2. **Спец 2 — game-session + WebSocket-сервер + комнаты (Слои 1–2).** Транспорт-агностичный
   сервис партии + WS-адаптер, коды комнат, реконнект.
3. **Спец 3 — React-фронт (Слой 3).** Отзывчивый стол, взаимодействия, кнопки
   ШУХ/голосование/«Одна карта».
4. **Позже.** Telegram-адаптер, опциональные варианты §12, аккаунты/история, серия партий
   (R-10.2), матч-статистика.

## 6. Локальная разработка

- **Git-хуки.** После клонирования один раз выполни `make install-hooks` — поставит
  версионируемые хуки из `scripts/git-hooks/` симлинками в `.git/hooks/`.
- **pre-commit.** Когда staged-изменения затрагивают `engine/`, `go.mod` или `go.sum`,
  хук гоняет `go build ./engine/...` и `go test ./engine/...`; коммит с падающей
  сборкой/тестами отклоняется. Обход при необходимости — `git commit --no-verify`.
